# 飞书协作指令

所有群内操作使用**固定关键词**，Review / 续做指令**必须携带任务号**，避免多任务时串单。

## 指令一览

| 操作 | 固定格式 | 说明 |
|------|----------|------|
| 创建任务 | `新建任务：需求描述` | 唯一创建入口，冒号后为需求正文 |
| Review 1 通过 | `#12 同意动工` | 仅当任务处于 Review 1 |
| Review 2 通过 | `#12 合并发布` | 仅当任务处于 Review 2 |
| 驳回重做 | `#12 驳回：你的意见` | Review 1 → 重新拆解；Review 2 → 重新开发 |
| 失败/中断续做 | `#12 续做` | 仅当任务为失败或已中断 |
| 取消任务 | `#12 取消任务` | 永久结束，不可续做（已完成除外） |

把示例中的 `12` 换成实际任务号。支持中文冒号 `：` 或英文冒号 `:`。

## 创建任务

```text
新建任务：测试登录后发布
验收标准：登录无异常
```

规则：

- 必须以 `新建任务：` 或 `新建任务:` 开头
- 冒号后的全部内容写入任务验收标准
- 第一行作为任务标题
- 创建后机器人会回复任务号，例如 `#13`
- **同一项目下若已有内容完全相同且未完成的任务，会提示「不可重复创建」并返回已有任务号**

**错误示例（不会创建）：**

```text
做一个登录需求
新建任务，测试登录
```

## 自动 Review（基于个人偏好）

当你用 `#N 驳回：意见` 留下 Review 意见时，系统会写入**该任务创建人**的偏好总结。  
下次进入人工 Review 1 / Review 2 **之前**，会先按这份偏好做自动 Review；飞书推送摘要里会附带「自动 Review 建议」（结论 / 说明 / 建议条目）。  
尚无历史意见时，自动 Review 会标记为跳过，但仍会进入人工确认。

## Review 1（需求拆解后）

机器人推送拆解摘要后，只接受：

```text
#13 同意动工
```

```text
#13 驳回：子任务需补充单元测试说明
```

**错误示例：**

```text
同意动工
#13 通过
驳回：写得不行
```

## Review 2（代码生成后）

```text
#13 合并发布
```

```text
#13 驳回：登录页样式需要调整
```

## 失败 / 中断后续做

```text
#13 续做
```

不要写 `重试`、`继续做`、不带任务号的 `续做`。

## 取消任务

永久结束任务（与「中断」不同：取消后不可续做）：

```text
#13 取消任务
```

执行中、Review 等待、失败、中断状态均可取消；已完成或已取消的任务不可再取消。

## 完整流程示例

```text
你：新建任务：实现看板搜索框
机器人：已创建任务 #13 …

（拆解完成后机器人推送 Review 摘要）

你：#13 同意动工
机器人：Review 1 已通过，已启动自动开发

（开发完成后推送 Review 2）

你：#13 合并发布
机器人：进入提交发布

# 若失败
机器人：任务执行失败 … 续做：#13 续做
你：#13 续做
```

## 注意事项

1. 一条消息只发一条指令，不要混写
2. 任务号前必须有 `#`
3. 指令与任务号以外不要加多余字（驳回意见写在 `驳回：` 后面）
4. **格式或状态不对时，机器人会回复「指令有误」**，并给出你发送的内容和正确写法

### 常见反馈示例

```text
指令有误
你发送：同意动工
格式不正确。
正确写法：#任务号 同意动工

可用指令：
· 新建任务：需求描述
· #任务号 同意动工
· #任务号 合并发布
· #任务号 驳回：意见
· #任务号 续做
· #任务号 取消任务
```

```text
指令有误
任务 #13 当前不能「同意动工」。
当前状态：Review 2，可用：#13 合并发布 / #13 驳回：意见 / #13 取消任务
```
